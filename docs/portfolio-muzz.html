UX Discovery, UX Design and UI Design : Muzz

Intro

As part of Muzz' interview process for a Product Design role, I was set a challenge to solve a complex UX problem. This comprised of product thinking, UX and UI design. The exercise tested fundamental principles of a typical UX/UI role; first I needed to understand Muzz' users & their needs, followed by crafting a considered and pixel-perfect user interface design. The challenge was deliberately open-ended to allow for maximum creativity.

The Brief

Muzz is a mobile app connecting Muslims worldwide through dating experiences. I was asked to wireframe and design a new feature around speed dating that would enable potential two matches to be paired for a short interaction (e.g. a video call) before giving them the ability to choose whether they want to match.

Research into the Domain

Approach

In order to fully understand the company, the market and identify the critical design problem that needed to be solved, I took the following approach:

  • Researched how other dating apps are supporting online speed dating and which of these have a "speed dating" feature already.
  • Downloaded Muzz and created an account to familiarise myself with the app, and consider where the new feature would be located.

Business Strategy and Growth

I also:

  • Identified the risks associated to the app.
  • Created plans for monetisation techniques, to ensure that this feature would be able to create new revenue streams for the business.
  • Decided on a name for the new feature.

Deliverables

I created the following:

  • UX: Low-fidelity wireframes to convey information architecture and to achieve the correct layout.
  • UI: High-fidelity designs to demonstrate how the new feature might look and feel, which conformed to the existing brand and app. The goal was to specify this to an appropriate level of detail such that an engineer would be able to build this without further clarification.
  • Prototype walkthrough to demonstrate the feature interactively.
  • Design System of components with instances/variants of each component, so that there was a single source of truth for each element.

I also wrote a report to explain how the new feature solves the design problem, detailing my thought process and design decisions made.
